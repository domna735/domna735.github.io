Cloud-Hosted Task Management Microservice System

Executive summary

COMP4442 Group 14 · last update: 2026-03-29.

Cloud Compute Service combines three capabilities in one coherent backend: session-based authentication, user-scoped task CRUD, and a utility compute API. The service is implemented as a layered Spring Boot application (Controller, Service, Repository, Entity) with security and validation controls across boundaries.

  • Authentication APIs: register, login, logout, and current-user retrieval.
  • Task APIs: create, list, get-by-id, update, and delete.
  • Compute APIs: service ping and arithmetic calculation endpoints.
  • Static frontend pages for register/login/task workflows.
  • Repeatable verification via integration tests and smoke scripts.

Architecture and security model

Layered design with ownership isolation by construction.

Controller → Service → Repository → Database

Auth identity source:
Spring Security SecurityContext (session-backed)

Ownership path:
Authenticated principal → service methods → user-scoped repository queries
  • Session authentication is handled by Spring Security; client state is carried by JSESSIONID cookie.
  • Passwords are stored as BCrypt hashes (never plaintext).
  • Task queries are ownership-constrained (for example: findByIdAndUserId) to prevent cross-user data access.
  • Protected APIs return 401 when unauthenticated; protected pages redirect to login.

Verification and reproducibility

  • Automated integration tests cover auth flow, protected routes, task CRUD lifecycle, and error semantics.
  • Smoke scripts provide fast API-level confidence checks before demos and pushes.
  • One-command development script starts the service, waits for readiness, and runs smoke validation.
  • Status-code alignment was enforced in scripts/documentation (register endpoint uses 201 Created).
Typical smoke check sequence 
Ping → Register (201) → Login → Current user
→ Create task (201) → List tasks → Logout

Cloud deployment readiness

  • Profile-based configuration separates local development and production runtime concerns.
  • Development profile uses H2 for fast local iteration.
  • Production profile externalizes datasource via environment variables (EC2 + RDS scenario).
  • Deployment assets include runtime templates, verifier scripts, and systemd service templates.

Limitations and next steps

  • Revisit CSRF strategy for broader production contexts.
  • Replace schema auto-update with migration tooling (Flyway or Liquibase).
  • Add login abuse protection (rate limiting, lockout policies).
  • Introduce pagination/filtering for task listing and larger datasets.
  • Expand observability with structured logs, metrics, tracing, and alerts.
